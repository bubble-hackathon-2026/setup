# Bubble Workflows — Design & UX Patterns

## Layout anatomy (3-panel shell)

The Workflows tab uses a rigid three-panel layout that sets the entire editor's rhythm:

- **Left sidebar (~200px, white bg)** — Workflow navigator. Tree of events grouped by type ("On this page", "An element is clicked", "Custom event"), each with a numeric count badge and expand/collapse carets.
- **Center canvas (flex, white bg)** — The actual flow, rendered as stacked cards connected by vertical arrows. Has its own header strip with the workflow name, view-mode toggle, and "Add notes" button, plus a zoom control pinned to the bottom-left.
- **Right properties panel (~310px, white bg)** — Dynamically populated for the selected node. Sections collapse with triangular carets. A BETA toggle + overflow (⋯) menu sit in its header.

The outermost chrome (top menu bar, left icon rail) persists from the Design tab — consistency across modes is the hidden strength.

## Core interaction vocabulary

What a user can do in the Workflows tab, expressed as UX capabilities:

**Manage events**
- Create a new event with "+ New" (primary cobalt button, top-right of the sidebar).
- Choose an event trigger from a 3-category picker: *General* (user logged in/out, page loaded, schedule, error), *Elements* (element-specific triggers), *Custom* (reusable named flows).
- Rename inline (the event name doubles as an editable textbox).
- Expand/collapse event groups via caret; each group shows a count badge.
- Filter/search events through the little filter icon at top of sidebar.

**Build a flow**
- Add a step by clicking the `+` button below the last step (or between steps on hover — inferred from the dashed insertion slot pattern).
- Pick an action from a two-level menu: 10 top-level categories (Account, Navigation, Data (Things), API Connections, Email, Payment, Analytics, Element Actions, Plugins, Custom Events, Backend Workflows), each revealing a flyout submenu.
- Search any action via the "Search for an action" input pinned at the top of the picker.
- Reorder steps via drag handle (⋮⋮) which appears on hover.
- Delete / duplicate / add notes via the ⋯ kebab menu that appears on hover.
- Switch between **flow view** (vertical cards with arrows) and **compact view** (dense numbered list) using the pair of icon toggles in the canvas header.
- Zoom in/out via the `— 100% +` pill at the bottom-left of the canvas.

**Configure a step**
- Clicking a step loads its properties into the right panel with a "Properties" and "Only when" section by default.
- Required empty fields show a subtle red/pink outline instead of a harsh error color.
- Toggle "Add a breakpoint in debug mode" on any step.
- Add a conditional expression via the "Only when" `Add` chip — opens the signature Bubble dynamic-data picker.
- "Expand ↖" link in the bottom-right of the condition area opens a large editor.

**Contextual help**
- The ⋯ header menu surfaces "Add notes", "Reveal element" (jumps to the triggering element in the Design tab), and "Tutorials" — embedded, non-intrusive learning.

## Signature patterns worth copying

**Step cards**
- Width ~336px, height 40px, 8px radius, background `#EBF3FF` (cobalt-20) when normal, cobalt-70 (`#0C29AB`) 1px border when selected.
- Two stacked labels: a tiny 10px/16px 600-weight uppercase-ish "Step N / Element Event" caption in gray-60 (`#8A8A8A`), and a 12px/16px 400-weight title in gray-80 (`#1A1A1A`).
- A soft square icon tile sits on the left (slightly darker blue-gray fill) to communicate action category.
- Vertical arrow connectors (↓) sit centered between cards with generous vertical whitespace.

**The action picker menu**
- White popover with 4px radius and the menu shadow (`0 2px 8px rgba(0,0,0,.10), 0 1px 2px rgba(0,0,0,.10)`).
- Search input always docked at top.
- Category rows have a 16×16 monochrome icon, text label, and a right-pointing caret.
- Hovered row fills with a subtle gray tint; submenu opens beside the parent, aligned to the row.
- Submenus contain flat lists (no deeper nesting) — two levels is the hard ceiling.

**Dynamic data picker** (the "Add" chip in conditions)
- Triggers a popover with a search field and sections with UPPERCASE gray labels: `DATA SOURCES`, `ELEMENTS`, `BREAKPOINTS ON THIS PAGE`, each with its own collapse affordance.
- Items are plain text rows — no icons — and single-click inserts them as a token.
- This is the connective tissue for the whole "no-code expression" feel.

**Property panel form grammar**
- Section headers are 12px/16px **700** weight gray-80 with a small triangular collapse caret on the left.
- Field labels are 12px/16px 400 weight gray-70 (`#525252`) sitting *above* the input — never inline.
- Inputs themselves have a 1px `#E6E6E6` border, 4px radius, transparent background, 4–8px padding. When filled they look almost flat — the editor relies on whitespace and label position for grouping rather than heavy borders.
- Inline-only empty states: placeholder `Add` chips with a dashed border look like a "tap to fill" affordance — used for required complex fields (Email to reset, Condition).
- Dropdowns use a chevron affordance and the same border treatment as inputs.
- Toggle switches (iOS-style, cobalt when on) for binary settings like breakpoints.

**Status & meta affordances**
- A small pill-shaped `BETA` badge in purple/cobalt tint sits beside the panel title — a reusable way to flag experimental surface area.
- A red triangle warning icon with a count (⚠ 1, ⚠ 2) in the top menu bar is the editor-wide issue tracker.
- The big top-right "Deploy" button is cobalt-70 — always visible, always reachable.

## UX principles the Workflows tab embodies

Bubble commits hard to a few ideas that any hackathon-friendly design system should replicate:

1. **Dense but breathable.** 12px type everywhere, 4–8px spacing as the default, but cards are padded generously (8–12px internal). It feels like IDE + design tool hybrid.
2. **Progressive disclosure through nested popovers.** A user never sees 50 options at once — two-level menus + search bars do the heavy lifting.
3. **Same visual language across modes.** The step card, the property panel, the picker menu — all share the same radius (4/8px), border (1px `#E6E6E6`), and shadow language.
4. **Empty-state as call-to-action.** Placeholders literally say "Add" in a dashed outline — no separate "create" button needed.
5. **Inline editing everywhere.** Workflow names, event names, and step titles are all double-click editable with the same text-input pattern.
6. **Color communicates category, not priority.** Cobalt = brand/primary, but purple for dynamic data, green for data operations, subtle grays for chrome — mapping color to semantic meaning rather than hierarchy.
7. **Escape + hover are first-class citizens.** Hover reveals drag handles, kebab menus, and insertion points; Escape dismisses any popover. The chrome stays quiet until you engage.

# Additional Bubble Editor Context

Focused on things not yet covered — cross-editor patterns surfaced during the Workflows research.

## Icon rail (far-left, 40px wide)
Persistent across all tabs. Vertical stack of 20×20 monochrome icons, ~24px vertical spacing, gray-60 default, cobalt-70 when active. Order top-to-bottom: Design (pencil), Workflow (nodes), Backend Workflows, Data (cylinder), API Connector ({}), Plugins, Logs, Settings, Analytics (bar chart), Security (shield). Bottom of rail: Help (?) and a user avatar circle. No labels — tooltips on hover only. This rail is the primary mode-switcher and never collapses.

## Top menu bar (44px tall, white bg, 1px bottom border `#E6E6E6`)
Left to right: Bubble logo (with dropdown caret), `⎇ Main` (version/branch switcher), `⌘ Web` (app selector), `📄 page_name` (page picker), contextual element selector (e.g. "Button change email is clicked"), then a spacer, then: issue count (⚠ red triangle + number), Preview (▶ split button with caret for options), Deploy (cobalt-70 primary button). All controls are 32px tall.

## Breadcrumb / contextual selector pattern
Each dropdown in the top bar is a button with an icon + label + caret (v). Click opens a searchable popover menu. The same pattern repeats in the sidebar ("ON THIS PAGE 2" with filter icon and expand icon). Consistency: anything selectable that has multiple options gets a caret.

## Badge system
- **Count badges**: small rounded-rect, 16–18px tall, cobalt-20 background, cobalt-70 text, 600 weight, 10–11px. Used next to event groups ("1", "2") and issue count (red-20 bg, red-70 text when errors).
- **Status pills**: BETA badge — periwinkle tint bg, periwinkle-80 text, 700 weight, 10px, fully rounded. Same shape for NEW, PRO, etc.

## Tooltips
Dark (`#1A1A1A`) background, white text, 12px/16px 400 weight, 4px radius, 6–8px padding, no arrow. Appears ~200ms after hover, positioned above/below target with 4px offset. Used heavily since the icon rail and canvas buttons are icon-only.

## Zoom control
Pill-shaped, bottom-left of canvas. White bg, 1px `#E6E6E6` border, 4px radius, contains `—` button, percentage text (e.g. `100%`), `+` button. Dividers between sections are 1px `#E6E6E6` verticals.

## Canvas background
Pure white `#FFFFFF`. No grid, no dot pattern, no gradient — just whitespace. The arrows between step cards are thin (~1px) gray-40 (`#C7C7C7`) vertical lines with a small arrowhead, roughly 32px long between cards.

## Loading / empty states
Empty workflow shows a large dashed "+ NEW EVENT" cobalt-30 outlined card centered in the canvas — same dashed-border language as the inline "Add" chips. The pattern: dashed borders always mean "this is empty, click to configure."

## Interaction states (across the editor)
- **Hover on list rows**: gray-10 (`#F7F7F7`) background fill, no border change, 100ms transition.
- **Selected list row**: cobalt-20 (`#EBF3FF`) background, cobalt-70 text, often with a 2px left border accent in cobalt-70.
- **Focused input**: border changes from `#E6E6E6` to cobalt-70, no box-shadow ring.
- **Disabled**: gray-40 text on gray-10 bg, no border, 0.6 opacity on icons.
- **Error/required**: red-40 (`#F9B3A9`) 1px border on the input, no red text label — the border alone communicates it.

## Button variants (measured)
- **Primary**: cobalt-70 bg, white text, 32px tall, 8px horizontal padding, 4px radius, 12px/16px 600 weight. No shadow.
- **Secondary**: white bg, 1px `#E6E6E6` border, gray-80 text, same dimensions.
- **Tertiary/ghost**: transparent bg, gray-70 text, no border, same dimensions. Hovers to gray-10 bg.
- **Icon button**: 28–32px square, transparent bg, gray-60 icon, hovers to gray-10 bg + gray-80 icon.
- **"+ New" button**: cobalt-70 bg, white text, small `+` icon on the left, used for primary creation actions in sidebar headers.

## Popover / menu conventions
- Always 4px radius, white bg, menu shadow.
- Min-width ~200px, max-width ~280px for action pickers.
- Item height 28–32px, 8px horizontal padding.
- Item layout: `[icon 16×16] [label] [optional caret/shortcut]` with 8px gap between.
- Section dividers: 1px `#F2F2F2` horizontal line, or small uppercase gray-60 text label (10px, 600 weight, letter-spacing normal) for categorized lists.
- Submenus open to the right, top-aligned to the parent item, 2px overlap.
- Close on: Escape, outside click, selection.

## Animation/motion
Very restrained. Panels don't slide. Popovers fade+scale in over ~100ms. Hover transitions are 100ms ease. The caret rotation on expand/collapse is 150ms ease. No bouncy easing anywhere — everything is flat and quick. This is part of the "feels like an IDE" identity.

## Typography hierarchy (consolidated)
```
Display / section title:       12px / 16px / 700 / gray-80
Body / control label:          12px / 16px / 400 / gray-70
Body emphasis (button, link):  12px / 16px / 600 / cobalt-70 or gray-80
Caption / metadata:            10px / 16px / 600 / gray-60  (uppercase-ish feel, no transform)
Large body (rare):             16px / 16px / 400 / gray-80
Code / expression tokens:      12px / 16px / 400 / monospace (fallback) — appear as pill-shaped chips
```
Note: Bubble doesn't really use H1/H2/H3 — the whole editor lives in the 10–16px range. If your design resource has heading styles, keep them small.

## Expression/token chips (critical Bubble pattern)
When dynamic data is inserted, it renders as an inline pill:
- Background: purple-20 (`#F6ECF8`) for dynamic data, cobalt-20 for static refs
- Text: purple-70 (`#79059A`) for dynamic, cobalt-70 for static
- Padding: 2px 6px
- Radius: 2px (small!) — intentionally less rounded than buttons
- Font: 12px 400, same as body
- Inline within text inputs, flowing with surrounding text

This chip is THE most recognizable Bubble UI primitive. If the hackathon resource needs one "signature" component, this is it.

## Drag & drop cues
- Drag handle icon: `⋮⋮` (6 dots in 2 columns), 12×16px, gray-40, appears on hover at the far-left of draggable rows.
- During drag: the row gets a subtle shadow and 0.9 opacity; a cobalt-70 1px horizontal line indicates the drop target.
- No ghost image customization — browser default with the shadow treatment.

## Keyboard shortcuts (observed behavior)
- Escape closes any popover/menu.
- Enter confirms inline edits.
- Cmd/Ctrl+Z / Shift+Cmd/Ctrl+Z map to the undo/redo buttons in the top bar.
- Cmd/Ctrl+F opens the "Find element" search.
- Tab moves between form fields in the properties panel.

## Accessibility notes
- All interactive elements have `aria-label` or visible text.
- Tree items use proper `role="treeitem"` with expand state.
- Focus rings are subtle (2px cobalt outline, 1px offset) — visible but not loud.
- Color contrast is solid: gray-70 on white passes AA; the cobalt-70 primary button passes AAA.

## Density / spacing rhythm
Bubble uses an 8px base grid but leans on 4px and 2px for tight clusters:
- Between sections in a panel: 16–24px
- Between fields in a section: 8–12px
- Between label and input: 4px
- Inside a card: 8–12px
- Between cards in canvas: 24–32px (plus the arrow)

## What's NOT in the Bubble UI (anti-patterns to avoid)
- No gradients anywhere in the editor chrome.
- No rounded-full pills (except badges and zoom control).
- No drop shadows on in-flow elements — only on floating surfaces.
- No dark mode (as of this session).
- No emoji in UI copy.
- No sentence-case toggling — labels are Title Case or sentence case consistently but never UPPERCASE except for section dividers in popovers.
- No skeuomorphic icons — everything is 1.5px stroke monochrome line icons, Lucide-style.

## Iconography system
Line icons, 1.5px stroke, 16×16 or 20×20, rounded line caps. Always monochrome, colored via `currentColor`. Categories of icons observed: navigation (chevrons, arrows), objects (database, envelope, credit card, chart), actions (plus, pencil, trash, eye), status (warning triangle, info circle, check). Very close to the Lucide or Phosphor icon libraries in style — if the other agent needs to pick an icon set, Lucide is the closest match.
