# Bubble Data Tab — Context for Your Other Agent

## Layout shift (important)

The Data tab **breaks the 3-panel pattern** from Design/Workflow entirely. It uses:

- Top horizontal sub-tab bar: **Data types | Privacy | App data | Option sets | File manager**
- Sub-tab bar sits directly under the top menu bar, ~44px tall, with 14px/14px 400 gray-80 labels (note: 14px is *larger* than the 12px used everywhere else in the editor — these tabs feel more prominent)
- Active sub-tab has a 2px cobalt-70 underline (same underline treatment as the Visual/Interaction/Conditional property tabs)
- Inactive sub-tabs share a 0.5px `#E6E6E6` bottom border with the active one (creating a single continuous underline)
- No left icon rail activity — the `Data` icon in the rail is the anchor, but the sub-tabs do all the navigation inside
- No right-hand properties panel in any sub-tab — Data is a full-width workspace, not an inspector-driven surface

Each sub-tab has its own distinct layout. I'll walk through them in order.

---

## 1. Data types sub-tab

**Layout:** Two-column: ~400px left master list, flex right detail panel.

**Left column — "Data types" master list**
- Section title "Data types" (14px/16px 700 gray-80)
- Search input (magnifier + "Data types" placeholder)
- List of data types, each row:
  - 32px tall
  - Type name in 12px 400 cobalt-70 when selected, gray-80 otherwise
  - Right-aligned gray-60 caption showing privacy status: `Publicly visible` or `Privacy rules applied`
  - Trash icon (16×16 gray-40, visible on hover)
  - Selected row: cobalt-20 (`#EBF3FF`) background, cobalt-70 text
- **"Bubble AI" card** below the list — distinct card treatment: 1px `#E6E6E6` border, 4px radius, ~8px padding, sparkle icon + "Bubble AI" label on left, cobalt-70 primary "Generate data types" button on right
- **"New type" creation card** at the bottom — 1px `#E6E6E6` border, 4px radius, stacked form:
  - "New type" label + text input (side by side)
  - "Make this data type private by default" with description "Things are visible to everyone" (a heading + helper row)
  - Cobalt checkbox on the right (but visually rendered right-aligned)
  - Gray-40 "Create" button bottom-right (activates to cobalt-70 when name is filled)

**Right column — "Fields for type X"**
- Section title "Fields for type {name}"
- "Type name" row: label + inline text input (editable) + notes bubble icon on the right
- **Fields table** — no visible borders, just row-based layout:
  - Column 1: field name (inline-editable text input)
  - Column 2: field type (shown as plain text e.g. "text", "date", "User")
  - Column 3: default value (shows "default" placeholder, or "Built-in field" caption in gray-60 italic for system fields)
  - Column 4 (hover only): notes icon + trash icon
- Built-in fields shown: **Creator** (type: User), **Modified Date**, **Created Date**, **Slug** — all marked "Built-in field" in gray-60
- User type has different built-ins: **email**, **Modified Date**, **Created Date**, **Slug**
- **"Create a new field"** ghost button at the bottom of the list (1px `#E6E6E6` border, cobalt-70 text)

**"Create a new field" modal**
- Centered modal, ~340px wide, white bg, 4px radius, `--b-lightbox-shadow`
- Title "Create a new field" (16px 700 gray-80, left-aligned)
- Two stacked labels: "Field name" (text input), "Field type" (dropdown)
- "This field is a list (multiple entries)" checkbox — single row with cobalt checkbox
- Primary cobalt "CREATE" button (note: uppercase label) + gray "Cancel" link, centered-ish in footer

**Field type dropdown** (inside the modal)
- Searchable dropdown with `Search...` input at top
- Categorized sections with uppercase gray-60 10px headers:
  - **DATA TYPES** — lists every custom data type in the app (User, Employee, Product, etc.)
  - **BASIC TYPES** — `text`, `number`, `numeric range`, `date`, `date range`, `date interval`, `yes / no`, `file`, `image`, `geographic address`
  - **PLUGIN TYPES** — types exposed by the API Connector (e.g. "Get IosSettings", "Make API Call", "Initialize API Call")
- Each section has a small collapse caret on the right of the section header

## 2. Privacy sub-tab

**Layout:** Same two-column as Data types. Left master list identical. Right panel shows "Privacy rules for type {name}".

**Empty state (right panel)**
- Centered title "Privacy rules for type X"
- Helper prose: "This type is visible by everyone. You can create a rule if you want to restrict access"
- Cobalt-outlined "Define a new rule" secondary button (white bg, 1px cobalt-70 border, cobalt-70 text) — note: outline style rather than filled

**Rule creation modal**
- Same centered modal shell, title "Define a new rule"
- Single field "Rule name" + text input
- Cobalt "CREATE" primary + "Cancel" link

**Rule card** (once created)
- Each rule is a ~700px wide card with 1px `#E6E6E6` border, 4px radius, `#FFFFFF` bg, 20px internal padding
- Header row: `Name` label + text input (the rule name), right-aligned notes bubble icon + trash icon
- `When` label + an expression field containing the Bubble **dashed "Add" chip** when empty, or a filled cobalt-70 token string when configured (e.g. `Current User is This User` rendered as a single cobalt-colored token)
- "Users who match this rule can..." subtitle (12px 400 gray-70)
- 2×2 grid of checkboxes:
  - `View all fields` · `Find this in searches`
  - `View attached files` · `Allow auto-binding`
- Multiple rule cards stack vertically

**"Everyone else (default permissions)" card**
- Always present at the bottom, below custom rules
- Same card treatment
- Same 4 checkboxes BUT `View all fields` expands to a per-field checkbox grid when unchecked:
  - Shows a 2-column indented list of every field on the type (api-test, Created Date, Modified Date, Slug, email, Social Networks, etc.) with individual checkboxes
  - This is the field-level permissions escape hatch
- Below the per-field grid: a separate cluster with `Find this in searches`, `View attached files`, `Allow auto-binding` as their own top-level checkboxes

**"Define a new rule"** button sits below the Everyone-else card (same cobalt-outlined style).

**Key UX pattern:** rule cascade is top-down — first matching rule wins, and Everyone-else is the fallback. No explicit indicator of this order in the UI (users infer from stack order).

## 3. App data sub-tab (database browser)

**Layout:** Two-column again but the left sidebar is narrower (~245px) and the right side is a full spreadsheet-style grid.

**Top strip** (spans full width)
- Page title: `Application data - {view name} - Development version` (12px 400 gray-80)
- Right-aligned links (red-70 `#B0200C` for emphasis):
  - `Copy and restore database` — opens modal to backup/restore
  - `Switch to live database` — toggle between Dev/Live data

**Toolbar row**
- Left: search input with magnifier icon + "Search for data entries" placeholder
- Then: entry count `N entries (displaying N)` (12px 400 gray-60, two-line)
- Right-aligned button cluster, each 32px tall, 1px `#E6E6E6` border, white bg, gray-80 text:
  - `New entry` (default enabled)
  - `Delete (0)` — count badge in parentheses, grayed out when nothing selected
  - `Upload` — CSV import
  - `Modify` — bulk edit selected
  - `Export` — CSV export
  - `Bulk` — bulk operations
- Note: these buttons look identical but perform very different actions — the entire Bubble admin pattern trusts labels over visual differentiation

**Left sidebar**
- Two buttons at top: `New view` and `Edit primary fields` (cobalt-outlined secondary style, side-by-side)
- Search input with "Views or data types" placeholder
- List of views:
  - Built-in views: `All Employees`, `All iossettings`, `All Products`, `All Stores`, `All test_dts`, `All Users`
  - Each row: view name (12px 400 gray-80, **bold** when selected), right-aligned copy-icon + pencil-edit-icon (16×16 gray-60)
  - Selected view: cobalt-20 background, cobalt-70 text
- Custom views are indented under the type they belong to (e.g. "All Users modified" appears indented under "All Users")

**Data grid**
- Header row: 12px 600 gray-80 labels on gray-10 (`#F7F7F7`) background
- Columns always start with: checkbox (select all) + icon column + data columns specific to the type
- For User: `Email | api-test | Created Date | Modified Date | Slug`
- Row layout:
  - Leftmost: selection checkbox + pencil icon (edit) + `Run as →` link (cobalt-70 with arrow) for User type specifically
  - Cells: 12px 400 gray-80 text, left-aligned, 8px vertical padding
  - Dates rendered as `Sep 30, 2025 10:17 am` (human-readable, not ISO)
- **`2 additional fields` link** above the grid right-aligned — indicates hidden columns, click to expand
- No visible vertical grid lines; horizontal 1px `#F2F2F2` separator between rows
- Row hover: gray-10 bg fill

**Bottom strip**
- Left: `Refresh data` link (cobalt-70)
- Right: `Load 50 more items...` link (cobalt-70) — pagination affordance; `No more items` caption when exhausted

**"Create a new database entry" modal**
- Full-viewport backdrop (not centered card — actually takes over the entire workspace area)
- Title at top center "Create a new database entry"
- `Type of thing` dropdown (first field, always present)
- Below, a **2-column grid** of all editable fields for that type, labels left-aligned at ~120px, inputs ~250px wide
- Built-in fields (Creator, Created/Modified Date) are not shown on create
- Footer: centered cobalt `CREATE` button + right-aligned `Cancel` link

**"Modify an existing database entry" modal**
- Same layout as Create, but:
  - Title: `Modify an existing database entry`
  - All fields pre-populated, INCLUDING built-ins: `Created Date`, `Modified Date` (editable!), `Unique id` (shown as a long ID like `1759241859688x2566151127208575`)
  - Primary button says `SAVE` instead of `CREATE`

**"Create a new database view" modal**
- Simple: Name + Type of things dropdown + CREATE/Cancel

## 4. Option sets sub-tab

Same two-column layout as Data types and Privacy.

**Left column**
- List of option sets with trash icons (no privacy captions — option sets aren't governed by privacy)
- Bottom creation card: `New option set` label + input + `Create` button (same pattern as new-data-type card, but without the private-by-default toggle)

**Right column — "Attributes and options for set X"**
Two stacked sections separated by a thin horizontal rule:

**Attributes section**
- Title: "Attributes (fields that describe each option)" — 12px 700 gray-80
- Table-like layout (no borders): each row has name | type | default/built-in marker + hover icons
- Every option set has a built-in `Display` attribute of type `text` (shown with "Built-in attribute" caption)
- `Create a new attribute` ghost button

**Options section**
- Title: "Options"
- One row per option: the option's Display value (editable text input) + `Modify attributes` gray-60 placeholder link (opens attribute editor) + notes + trash icons on the right
- Bottom: `New option` label + input + gray `Create` button (activates on input)

**Mental model:** Option sets = lightweight enums with extensible per-option metadata. The `Display` attribute is the label; you can add typed attributes (e.g. color, sort order, icon) that travel with each option.

## 5. File manager sub-tab (brief)

**Layout:** Full-width, single-panel.

- Title: "File manager" + right-aligned red-70 `Switch to live database` link
- Filter bar: `File name` input + `File type` dropdown (default "All files") + `Only private files` checkbox
- Right-aligned actions: `Search` (cobalt-outlined), `Delete` (grayed until selection), `Upload`
- `Showing files N - N of N` caption below filters
- Data grid columns: checkbox | icon | `File name` | `Size` | `Type` | `Upload date ▼` (with sort triangle) | `User ID` | `Attached to`
- Size formatted human-readable (`84.7 KB`), type is MIME (`image/jpeg`), dates match App data format
- "Attached to" column links to the data thing that references this file (blank if orphaned)
- Footer: `No more items` or `Load more` pagination

## Global UX patterns specific to the Data tab

**Dual environment (Dev/Live)**
The red-70 `Switch to live database` / `Copy and restore database` links are the critical escape hatch. Red coloring communicates "this affects production data" — it's deliberately alarming compared to the gray and cobalt of normal actions. This appears on App data and File manager tabs but not on schema-editing tabs (Data types, Privacy, Option sets are schema and deploy with the app, not data).

**Schema vs. content separation**
- Data types / Privacy / Option sets = schema (deployed with the app)
- App data / File manager = runtime content (separate Dev/Live environments)
The sub-tabs sit in the same bar but the mental model shifts between them.

**Master-detail repetition**
Three of the five sub-tabs (Data types, Privacy, Option sets) use the identical two-column master-detail pattern. Consistency is the strength — once a user learns one, they know all three.

**Inline editing everywhere**
Type names, field names, rule names, option values — all double-click or single-click editable text fields rendered directly in place. No dedicated "rename" modal anywhere in schema editing. Changes auto-save on blur.

**Bubble expression tokens**
The `When` field on privacy rules uses the same cobalt-colored expression tokens seen in Workflow conditions. Empty state: dashed "Add" chip. Filled: inline cobalt-70 tokens that concatenate to read like natural English (`Current User is This User`). Clicking opens the full dynamic-data picker.

**Notes system (consistent pattern)**
Every major entity — type, field, rule, option — has a notes bubble icon. Clicking opens a right-side drawer:
- 450px wide, slides in from the right with white bg + shadow
- Header: `Notes` title + close X
- Subheader breadcrumb: `Field - iossettings - field_1` (context-aware, 12px 400 gray-60, with a small icon prefix)
- Sub-subheader: `This note hasn't been updated yet` (italic gray-60)
- `View all notes (N)` link in the header right
- Full-height textarea with placeholder `Add a note here...`
- Footer hint: `⊘ Notes save automatically` (gray-60)

**Confirmation patterns**
- No undo toast after destructive actions — trash icons typically trigger a browser `confirm()` dialog
- Bulk operations (Delete with multi-select) show the count in the button label: `Delete (3)` to reinforce what will be affected
- Schema changes are immediate; data changes (via Modify/Delete/Bulk) affect the selected environment only

**"Run as →" concept** (User type App data)
Each user row has a `Run as →` link that launches a preview session as that user — a distinctive affordance for auth testing. This pattern doesn't appear on other types. Style: cobalt-70 text + right-arrow glyph, no underline until hover.

**Privacy indicator chips in the master list**
The two-state caption (`Publicly visible` vs `Privacy rules applied`) acts as a quick-scan indicator — gray-60 12px text, no background. When all fields are in "Publicly visible" state, the developer knows they have no row-level security. This is an unusually honest UX pattern — it doesn't hide or soften the fact that data is open.

## Signature visual patterns (new in Data tab)

**Bubble AI promotion card**
A distinct design: 1px `#E6E6E6` border, 4px radius, small internal padding, a sparkle-wand icon + `Bubble AI` label in 12px 600 gray-80 on the left, and a cobalt-70 filled `Generate data types` button on the right. This is the ONLY place in the data tab where AI generation is offered, and it's visually marked as an "add-on" rather than part of the core create flow. The new-type form sits immediately below and looks similar in shell but different in chrome — the AI card has no gray-10 background while the new-type card does (subtle contrast).

**Red-70 destructive environment links**
Pure hypertext-style red links with no button chrome. Used exclusively for environment/database actions. Never used for element deletion (those use trash icons) or form cancels (those use gray "Cancel" text). This communicates: "this is an admin-level action that affects data stores, not UI state."

**Privacy rule "When" chip**
The expression field looks like a flat input but its border is lighter than a standard input (`#F2F2F2` instead of `#E6E6E6`) and it has slightly more vertical padding. Empty state shows a dashed "Add" chip centered inside it. This is the "drop zone for expressions" pattern, reused from Workflow conditions.

**2-column form grid** (in Create/Modify Entry modals)
All entity edit forms use a two-column layout: label left (~120px, right-aligned), input right (~250-300px). Fields flow left-to-right then wrap. This pattern is unique to Data's modals — Workflows and Design use top-aligned labels. The difference signals "this is a form for database records, not UI configuration."

**Field-level permission grid**
When "View all fields" is unchecked in a privacy rule, the card expands to reveal a 2-column checkbox grid of every field with 8px row spacing and no visual grouping between related fields (created_date sits next to slug). It's dense but functional — deliberately raw to expose the security model without hiding complexity.

## Key UX principles specific to Data tab

1. **Schema is code, data is content.** The sub-tab ordering (types → privacy → data → option sets → files) reflects a "build the schema first, then manage what lives in it" workflow. Privacy sits intentionally between the two to make it impossible to ignore.

2. **Dangerous actions are always red, never hidden.** Switch to live database, Copy and restore, and field-level permission unchecks are all visually loud. Bubble treats its database as a production surface even during development.

3. **Built-in fields are informational, not editable.** The gray-60 italic "Built-in field" marker communicates "this is managed for you." Users can't delete Creator, Created Date, Modified Date, or Slug. This is a hard boundary visible in the UI.

4. **Privacy is opt-in with public default.** New data types default to "Publicly visible" unless the checkbox is flipped. The privacy indicator in the sidebar makes this visible at a glance. The philosophy: trust developers to tighten, don't force them up-front.

5. **The grid is the record.** App data uses a real spreadsheet pattern (rows, columns, sort, search, bulk select) because that's the mental model users already have for databases. No custom abstractions, no cards — just tabular data.

6. **Notes are universal.** Every significant entity gets a notes drawer. This is documentation-in-place, meant for team members to understand schema intent later. The drawer's breadcrumb title makes clear which entity is being annotated.

7. **Modals lock the full workspace.** Unlike Design-tab popovers that float alongside the panel, Data tab modals take over the full viewport (Create/Modify entry) or center with a dimmed backdrop (Create type, Create rule, Create view). This reinforces that data operations are transactional and should not be casually interleaved with other work.

That should give your other agent everything to build data-tab components and context. Key takeaways for any "looks like Bubble Data" implementation: horizontal sub-tabs at 14px with underline, master-detail 400px left / flex right, spreadsheet-style app-data grid, red-70 destructive environment links, cobalt expression tokens for privacy rules, 2-column label-left form modals, and notes drawers on every entity.
